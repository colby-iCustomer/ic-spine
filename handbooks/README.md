# handbooks/

How each team operates at iCustomer. Audience-specific operational knowledge — the stuff a new hire reads to understand "how we do things here."

## Structure

```
handbooks/
├── engineering/      (will hold icustomer-docs/engineering content — SDLC, CI/CD, ADRs, agent taxonomy)
├── product/          (sprint cadence, ticket schemas, PM rituals)
└── marketing/        (lifecycle emails, launch playbook, content operations)
```

`customer-success/` will be added after the PM stream proves the model works.

## What handbooks are NOT

- **Architecture** lives in [architecture/](../architecture/). Handbooks tell you HOW we work; architecture tells you WHAT the system is. Cross-reference, don't duplicate.
- **Product positioning** lives in [products/](../products/). Each product has its own positioning + state.
- **Templates** live in [templates/](../templates/). Handbooks reference templates, they don't contain them.

## Update cadence

Cold — quarterly-ish. If a handbook page changes more than once a quarter, the underlying process is unstable, not the doc.
